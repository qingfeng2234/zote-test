# AI 时代的学习范式与教学范式总结

> 来源：李勃老师课程网站（https://李勃老师.com/）及博客文章《如何做好一门技术课程》《效率和用心》

---

## 一、核心定位：不是讲知识，而是设计学习路径

传统教学假设“老师懂得多 = 课程好”，而这个范式认为：

> **课程质量的上限，不由知识储备决定，而由学习路径设计决定。**

学习者的目标不是参观讲师的知识库，而是完成一次能力迁移：

> 从“不会” → 到“会一点” → 再到“能独立做出来”。

因此，这套范式的核心是**目标驱动的路径工程**：以学习者的终点能力为锚点，倒推必要内容、顺序和实践。

---

## 二、关键教学原则

### 1. 结果倒推，而非知识树顺讲
- 先定义学习者最终能独立完成什么（如“部署出第一个 Web 应用”）。
- 围绕这个结果反推必要能力，而不是把讲师熟悉的知识树全部搬一遍。
- 好处：内容聚焦、无关扩展被克制、路径更像“能走通的路”。

### 2. 明确人群，敢于划定边界
- 做课程前先回答“给谁学”。
- 需要明确：
  - 学习者目前已知什么、未知什么；
  - 主要障碍是技术难，还是路径/工具复杂；
  - 学完后要能独立完成什么。
- “面向所有人”等于“没有面向任何人”。边界越清晰，主线越稳定。

### 3. 顺序设计优先于内容覆盖
- 很多课程失败不是因为内容错了，而是因为顺序错了。
- 推荐的认知顺序：
  1. 先给场景；
  2. 再给最小可运行结果；
  3. 再解释原理；
  4. 最后逐步增加复杂度。
- 学习顺序不是“知识树怎么长”，而是“人怎么吸收”。

### 4. 做中学，但反对作业堆砌
- 技术能力本质是操作、判断、排错、迁移，课程必须有实践任务和阶段产出。
- 但“实践驱动”不等于“作业堆满”。
- 每个练习都要服务能力目标，让学习者每学完一个阶段都能回答：

> **“我现在具体能做什么？”**

### 5. 主动管理挫败感
- 新手最怕的不是难，而是不确定。
- 课程设计要前置处理：
  - 预判高频卡点；
  - 给出最小排错路径；
  - 明确“哪些报错是常见且正常的”；
  - 把任务拆成可获得反馈的小步。
- 目标不是“提高热情”，而是“避免放弃”。

### 6. AI 时代的角色转换：从写代码到管理代码
- 核心假设：代码可以由 AI 生成，学习者的职责是理解原理、组织和运行代码。
- 学习者要从“实现者”转向“代码的组织者和管理者”。
- 掌握概念与术语，用清晰需求指导 AI，比记忆语法更重要。

## 三、这种范式对“学什么”的回答

对初学者来说，最重要的不是“更多知识”，而是：

| 初学者真正需要 | 这套范式的处理方式 |
|---|---|
| 降低不确定感 | 先给场景和可运行的最小结果 |
| 建立信心 | 每个阶段都有可见产出 |
| 知道自己在哪 | 清晰的学习路径和进度感 |
| 遇到错误不崩溃 | 预判卡点，给排错路径 |
| 能独立完成东西 | 终点倒推，围绕真实项目组织内容 |

因此，它教的通常是**能用的知识**：每个知识点都对应一个真实动作，而不是孤立的理论概念。

## 四、可迁移到科研与学习的方法

这套范式不只适用于编程教学，也可以迁移到硕士研究、论文写作和复杂任务管理中：

| 教学原则 | 在科研/学习中的对应 |
|---|---|
| 终点倒推 | 先定论文要回答的科学问题，再反推实验设计 |
| 明确人群 | 明确论文读者，判断哪些背景要讲、哪些不用讲 |
| 克制而完整 | 砍掉“测了但用不上”的指标，只保留能回答问题的实验 |
| 做中学 | 每轮实验都对应一个可验证的小结论 |
| 管理挫败感 | 提前写 SOP、检查表，知道卡住时怎么回退 |
| AI 时代重理解 | 数据处理/作图可用工具，但核心假设和机制必须自己懂 |

### 应用于硕士项目的具体思路

1. **用终点倒推梳理论文主线**
   - 先写核心结论句：
     > “PLA/PS 来源的 DOM 通过改变微生物可利用碳源和降解菌活性，影响 PHE 降解效率并改变代谢产物组成。”
   - 再反推需要哪些数据支撑：PHE 降解曲线、DOM 特征、代谢产物、微生物指标。
   - 区分“必要且充分”与“可有可无”的实验。

2. **把每轮实验设计成可交付的小阶段**
   - 每轮实验结束后回答：
     > **“这轮实验结束后，我具体知道了什么？”**
   - 例如：
     - 第一轮预实验：确认 DOM 和 PHE 浓度设置合理；
     - 第二轮预实验：比较 PLA/PS/沉积物 DOM 的影响差异；
     - 正式实验：验证差异并补充机制。

3. **用 SOP 和检查表管理挫败感**
   - 把常见失败点写进 SOP（如 MPs 悬浮、代谢产物浓度低、OD600 偏低）；
   - 给每个操作“最小排错路径”：如果 X 发生，先检查 Y，再检查 Z；
   - 每轮实验后强制填写检查表，确保数据可靠、问题记录、下一轮明确。

4. **论文写作按“场景-结果-原理”顺序**
   - 场景：为什么要研究这个问题？
   - 最小结果：核心发现是什么？
   - 原理解释：为什么会这样？
   - 逐步扩展：与文献对比、机制推测、不足与展望。

## 五、可进一步学习的内容

如果想深入学习这种教学/学习方式，可以看：

- **认知负荷理论**（Cognitive Load Theory, Sweller）：解释为什么顺序比内容更重要。
- **逆向课程设计**（Understanding by Design, Wiggins & McTighe）：从预期结果倒推教学设计的经典框架。
- **观察优秀课程结构**：把“零到全栈”课程的模块表拆成“目标-内容-实践-反馈”四列，模仿其结构。
- **小范围实践**：下次组会汇报、带本科生做实验、写 SOP 时，尝试用“终点倒推 + 阶段交付 + 管理卡点”的思路。

---

## 六、一句话总结

> **做任何复杂的事之前，先想清楚：谁会用到这个结果？他们现在在哪？怎么让他们不卡住地走完全程？**

这种思维方式适用于教学、科研、写作和项目管理。
